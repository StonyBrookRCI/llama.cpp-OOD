# llama.cpp Open OnDemand App

This [Open OnDemand (OOD)](https://www.openondemand.org/) interactive app launches [**llama-server**](https://github.com/ggml-org/llama.cpp) (from llama.cpp) on an HPC compute node with GPU acceleration. It provides a browser-based chat interface and an OpenAI-compatible API for running large language models locally using GGUF model files. Users connect directly through OOD's reverse proxy; no SSH tunnel required.

## What this app provides

- A point-and-click OOD form to request GPUs, memory, walltime, and choose a model from a dropdown.
- A single Slurm job that starts `llama-server` bound to `0.0.0.0` on a random port.
- A custom OOD view page with a "Connect to LLM Chat" button that opens the chat UI through the reverse proxy (`/rnode/`).
- Built-in Help & Documentation window with model download instructions, troubleshooting, and admin guide.

## Directory layout

- `manifest.yml`. OOD app metadata.
- `form.yml.erb`. The user-facing form (queue, GPUs, model selection, context size, etc.).
- `submit.yml.erb`. Slurm submission parameters.
- `template/before.sh.erb`. Picks a free port, resolves hostname, writes `.host` and `.port`.
- `template/script.sh.erb`. Starts `llama-server` with the selected model and writes `connection.yml`.
- `template/connect.yml.erb`. Template for OOD to read host and port.
- `template/after.sh.erb`. Post-run hook.
- `view.html.erb`. Renders the Connect button and Help & Documentation window.
- `icon.png`. App icon.
- `module/llama.cpp/8184`. Reference TCL module file.

## Prerequisites

### OnDemand requirements

- Open OnDemand with **Batch Connect** enabled.
- A working **Slurm** integration for OOD (the app uses Slurm `--partition`, `--mem`, and `--gres=gpu:*`).
- **Reverse proxy** enabled. The app uses `/rnode/<host>/<port>/` URLs. Your `ood_portal.yml` must have `rnode_uri` configured. See [Update the reverse proxy URL](#5-update-the-reverse-proxy-url) for details.

### Software requirements on compute nodes

- **llama.cpp** compiled from source with CUDA support. See [Building llama.cpp](#building-llamacpp) below.
- `python3`. Used to allocate free ports.
- `curl`. Used for server readiness checks.

### GPU and driver requirements

- NVIDIA GPUs with compatible drivers and CUDA runtime for your llama.cpp build.
- The build at our site targets compute capability, e.g `120` (NVIDIA RTX PRO 6000 Blackwell). Adjust `CMAKE_CUDA_ARCHITECTURES` for your hardware.

### Storage requirements

The app reads model files. Make sure the following paths (or your replacements) exist and are accessible.

- System models directory (admin-managed, readable by all users): `/lustre/nvwulf/software/llama.cpp/models/`
- User models (optional): users can download their own GGUF files to scratch space (e.g., `/lustre/nvwulf/scratch/$USER/models/`).

## Building llama.cpp

llama.cpp must be compiled from source on your cluster to get native CUDA GPU support. Pre-built binaries and AppImages from other projects (such as Jan AI) typically require newer glibc versions than HPC nodes provide.

### Clone the repository

```bash
cd /lustre/nvwulf/software
git clone https://github.com/ggml-org/llama.cpp.git llama.cpp-src
cd llama.cpp-src
```

### Load build dependencies

```bash
cuda12.8/toolkit/12.8.0    # adjust to your cluster's CUDA version
```

Verify:

```bash
nvcc --version
```

### Configure and build

For most HPC nodes with older binutils, disable native CPU detection to avoid assembler errors with newer instruction sets (AVX-VNNI, `vpdpbusd`):

```bash
cmake -B build \
  -DGGML_CUDA=ON \
  -DCMAKE_BUILD_TYPE=Release \
  -DBUILD_SHARED_LIBS=OFF \
  -DCMAKE_CUDA_ARCHITECTURES="90;120" \
  -DGGML_NATIVE=OFF \
  -DCMAKE_C_FLAGS="-march=x86-64-v3" \
  -DCMAKE_CXX_FLAGS="-march=x86-64-v3"

cmake --build build --config Release -j
```

**Build flags explained:**

| Flag | Purpose |
|------|---------|
| `-DGGML_CUDA=ON` | Enables NVIDIA GPU support |
| `-DBUILD_SHARED_LIBS=OFF` | Static linking, no shared library issues |
| `-DCMAKE_CUDA_ARCHITECTURES="90;120"` | Target GPU compute capability. Common values: `90` (H200), `120` (Blackwell B40). Use `nvidia-smi` to check yours. |
| `-DGGML_NATIVE=OFF` | Disables `-march=native` which can emit instructions that older assemblers cannot handle |
| `-DCMAKE_C_FLAGS="-march=x86-64-v3"` | AVX2+FMA optimization without triggering assembler errors |

> **Troubleshooting build errors:**
>
> - `CUDA_ARCHITECTURES is set to "native", but no GPU was detected` — You are building on a login node without GPUs. Specify the architecture explicitly with `-DCMAKE_CUDA_ARCHITECTURES="120"`.
> - `Error: unsupported instruction 'vpdpbusd'` — Your binutils is too old for AVX-VNNI instructions generated by `-march=native`. Add `-DGGML_NATIVE=OFF` and `-DCMAKE_C_FLAGS="-march=x86-64-v3"`.

### Install binaries

```bash
mkdir -p /lustre/nvwulf/software/llama.cpp/8184/bin
cp build/bin/* /lustre/nvwulf/software/llama.cpp/8184/bin/
chmod a+rx /lustre/nvwulf/software/llama.cpp/8184/bin/*
```

The `8184` corresponds to the llama.cpp build number (shown at startup). Adjust as needed.

### Verify

```bash
/lustre/nvwulf/software/llama.cpp/8184/bin/llama-server --help
```

## Creating the environment module

Create a TCL module file at your modules path. A reference file is included in `module/llama.cpp/8184`:

```tcl
#%Module1.0
proc ModulesHelp { } {
    puts stderr "llama.cpp - Local LLM inference engine"
    puts stderr "  llama-server  - Web UI + OpenAI-compatible API server"
    puts stderr "  llama-cli     - Command-line chat interface"
    puts stderr "System models: \$LLAMA_SYSTEM_MODELS"
}

module-whatis "llama.cpp - LLM inference engine with CUDA support (build 8184)"

set     llama_root      /lustre/nvwulf/software/llama.cpp/8184
set     llama_models    /lustre/nvwulf/software/llama.cpp/models

setenv  LLAMA_ROOT          $llama_root
setenv  LLAMA_SYSTEM_MODELS $llama_models
prepend-path PATH           $llama_root/bin
```

Place this file at your module path (e.g., `/lustre/nvwulf/software/modules/llama.cpp/8184`) and verify:

```bash
module load llama.cpp/8184
which llama-server
llama-server --help
```

## Setting up system models

Download GGUF models to the system directory so all users can select them from the form dropdown:

```bash
mkdir -p /lustre/nvwulf/software/llama.cpp/models
cd /lustre/nvwulf/software/llama.cpp/models

# Example: Mistral 7B
wget https://huggingface.co/TheBloke/Mistral-7B-Instruct-v0.2-GGUF/resolve/main/mistral-7b-instruct-v0.2.Q4_K_M.gguf

# Set permissions so all users can read
chmod a+r /lustre/nvwulf/software/llama.cpp/models/*.gguf
```

Then add the model to the dropdown in `form.yml.erb`:

```yaml
model_choice:
  options:
    - ["Mistral 7B Instruct v0.2 (Q4_K_M, ~4 GB)", "/lustre/nvwulf/software/llama.cpp/models/mistral-7b-instruct-v0.2.Q4_K_M.gguf"]
    - ["Llama 3.2 3B Instruct (Q4_K_M, ~2 GB)", "/lustre/nvwulf/software/llama.cpp/models/Llama-3.2-3B-Instruct-Q4_K_M.gguf"]
    - ["No - Use my own model", "custom"]
```

When you download a new model, just add one line to the `options` list.

## How it works

1. **before.sh** selects a random free port using Python's socket binding and writes it into `.port`. Resolves the short hostname via `hostname -s` and writes it to `.host`.

2. **script.sh**
   - Loads the `llama.cpp/8184` module.
   - Reads the model choice from the form. If the user selected a system model, uses that path directly. If "No - Use my own model" was selected, uses the user-provided custom path instead.
   - Validates the model file exists.
   - Writes `connection.yml` with the short hostname and port for OOD's reverse proxy.
   - Starts `llama-server` with `--host 0.0.0.0` (required for reverse proxy access from the OOD head node), the selected model, context size, and GPU layer count.
   - Waits up to 120 seconds for the server to become healthy via the `/health` endpoint.

3. **connect.yml.erb** and **view.html.erb**
   - `connect.yml.erb` produces `connection.yml` used by the view page.
   - `view.html.erb` reads `connection.yml` and renders a "Connect to LLM Chat" button linking to `https://<ood-host>/rnode/<host>/<port>/`. Also provides a "Help & Documentation" button that opens a full help window with getting started guide, model download instructions, admin guide, and troubleshooting.



## Troubleshooting

- **Job starts, but the page keeps refreshing.** Model loading can take time, especially for large models (70B+). Wait 1-2 minutes. Check `llama-server.log` in the job output directory.
- **"Connect" opens a blank page or error.** The model may still be loading into GPU memory. Refresh after 30 seconds. Check `llama-server.log`.
- **Model not found at launch.** Verify the GGUF file path is correct and the file exists. Only `.gguf` format is supported. Ollama models, PyTorch, or safetensors won't work.
- **Slow responses.** Ensure GPU layers is set to "All" and you requested enough GPUs for your model size.
- **Out of memory.** Try a smaller quantization (Q4_K_M instead of Q8_0) or request more GPUs.
- **Job fails immediately.** Check `llama-server.log`. Verify the module loads correctly: `module load llama.cpp/8184 && which llama-server`.
- **Port already in use.** The app picks a random free port. If this error occurs, relaunch the job.
- **Cannot reach the Connect URL.** Verify `/rnode/` is enabled in your OOD configuration. Test with a simple HTTP server as described in [step 5](#5-update-the-reverse-proxy-url).
